# Plan: Replace Notion Poll with Webhook

## Problem
The current `build_notion_to_slack_graph()` uses polling (`poll_node`) to detect Notion changes, which is inefficient.

## Solution
Replace polling with a webhook-triggered approach. Since Notion doesn't have native webhooks, we'll create a webhook endpoint that external services can call.

## Implementation Steps

### 1. Create webhook endpoint (FastAPI)
- Add `/webhook/notion` endpoint in `main.py`
- Accept POST requests with Notion change payloads
- Immediately invoke the graph asynchronously

### 2. Remove polling loop
- Remove the `lifespan` asyncio loop in `main.py`
- Delete `poll_node` or mark as unused

### 3. Add webhook node in nodes.py
- Create `webhook_node(state: SlackSyncState) -> SlackSyncState`
- Parse incoming webhook payload
- Set `state.notion_changes` directly from payload

### 4. Update graph in graphs.py
- Replace `"poll"` node with `"webhook"` node
- Update entry point from `"poll"` to `"webhook"`
- Simplify: no `route_changes` conditional needed (webhook fires only on changes)

### 5. Webhook payload format
External service sends:
```json
{
  "changes": [
    { "id": "page-id", "last_edited_time": "2024-..." }
  ]
}
```
- `webhook_node` extracts pages from `state.webhook_payload["changes"]`
- Falls through to `format_node` → `notify_node`

## Integration

**Chosen approach**: Event-driven only, external sends data

Since Notion lacks native webhooks, use an external service (Zapier/Make) to:
1. Watch Notion database for changes
2. POST to `/webhook/notion` with `{ "changes": [...] }`

The endpoint will:
- Accept POST with JSON payload: `{ "changes": [...] }`
- Skip if payload empty
- Immediately trigger graph for "format" → "notify" flow
